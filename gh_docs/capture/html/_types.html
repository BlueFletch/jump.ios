Mapping Reference

The Capture schema, the JSON language, Objective-C, and the JUMP for iOS libraries all use different nomenclature when referring to types. For example, the schema may define a property as having type "string", which is the same as the JSON "String" and the Objective-C foundational class "NSString".

Some types do not perfectly map between systems. For example, a boolean on Capture can have the values true, false, and null, while an NSBoolean in Objective-C can only be YES or NO. On the other hand an NSNumber can represent true, false, and null, but it can also hold many other things that won't be accepted by the Capture API (such as -100).

To ensure that your objects can correctly update themselves on Capture, we had to use an NSNumber to hold boolean properties. To make it perfectly clear that boolean properties cannot be just any number, we typedeffed them JRBoolean. Your IDE's auto-complete should suggest JRBoolean when setting a boolean property, but since underneath it is just an NSNumber, you can set it to null as well.

The compiler won't stop you from setting a JRBoolean to a non-boolean value, but you will get an error when you try and update your object on Capture.
