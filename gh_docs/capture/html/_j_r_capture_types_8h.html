typedef NSNumber JRBoolean

Some types do not perfectly map between Objective-C and Capture. For example, a boolean on Capture can have the values true, false, and null, while an NSBoolean in Objective-C can only be true or false. On the other hand, an NSNumber can represent true, false, and null, but it can also hold many other things that won't be accepted by the Capture API (such as -100) when the property has the type boolean.

To ensure that your objects can correctly update themselves on Capture (i.e., contain null), we use an NSNumber to hold boolean properties. To make it perfectly clear that boolean properties cannot be just any number, we typedeffed them JRBoolean. Your IDE's auto-complete should suggest JRBoolean when setting a boolean property, but since underneath it is just an NSNumber, you can set it to null as well.

When setting a property of type JRBoolean, you can set it as you would an NSNumber. Additionally, for your convenience, Capture objects that have boolean properties will provide getters and setters that accept and return primitive booleans. For example, the JRPhoneNumbersElement.primary property has the following getters/setters: JRPhoneNumbersElement.primary(), JRPhoneNumbersElement.setPrimary(), setPrimaryWithBool: (JRPhoneNumbersElement), and getPrimaryBoolValue (JRPhoneNumbersElement).

Note:
In Capture there is a difference between a boolean with the value false and a boolean with the value null.
Warning:
The compiler won't stop you from setting a JRBoolean to a non-boolean value, the JUMP for iOS library will automatically convert your JRBoolean properties to the equivalent boolean value when updating the object on Capture.

typedef NSObject JRJsonObject

Some types do not perfectly map between Objective-C and Capture. For example, a property of type json on Capture can hold any valid JSON object (objects, arrays, strings, numbers, booleans, or null). When deserializing properties of type json, the JUMP for iOS library stores the result in an NSObject property. That is, a json property can hold any object hierarchy that JSONKit can serialize. This includes and is limited to NSArray, NSDictionary, NSString, NSNumber, and NSNull.

For more information on JSONKit and what it can serialize check the README in the project: https://github.com/johnezang/JSONKit

To ensure that your objects can correctly update themselves on Capture, we use an NSObject to hold json properties. To make it perfectly clear that json properties can only be types that are accepted by JSONKit, we typedeffed them JRJsonObject. Your IDE's auto-complete should suggest JRJsonObject when setting a json property, but since underneath it is just an NSObject, you can set it to null as well.

Note:
If the JSON returned by Capture is an object, this property will hold an NSDictionary. If the JSON returned by Capture is an array, this property will hold an NSArray. If it is a string, NSString, number or boolean, NSNumber, and if the property is null, it will hold nil.
Warning:
The compiler won't stop you from setting a JRJsonObject to a value that may not be accepted by Capture or JSONKit, but you will get an error when you try and update your object on Capture.

typedef NSArray JRStringArray

Plurals contain collections of elements, where an element is just an object stored in a plural. Like objects, plural elements can contain primitive values, sub-objects, or sub-plurals. Plurals only store one type of element, and that element’s attributes are defined in the schema. Some plurals just store lists of strings. For example, the JRCaptureUser.profiles plural stores JRProfilesElements, and the JRProfile.quotes plural stores a list of strings (quotes).

For both types of plurals (plurals with strings and plurals with objects) use NSArrays to hold their elements. When the Capture User Model is generated, the script will determine if the plural property contains a list of strings or a list of elements.

If the plural only contains a list of strings, this property is typedeffed to JRStringArray. JRStringArray properties should only contain NSStrings. Your IDE's auto-complete should suggest JRStringArray when setting a plural of strings. For example, the JRProfile.books array and the JRProfile.heros array both holds a lists of strings (books and heroes, respectively).

If the plural contains elements that are more than just a string, a sub-object is created for that element. For example, the JRCaptureUser contains an NSArray property called JRCaptureUser.statuses. This array contains objects of type JRStatusesElements.

When adding elements to an array, you should only add the elements that belong in that array.

Warning:
The compiler won't stop you from setting a most objects to arrays. For example, you could add an NSDictionary to a JRStringArray, which is only supposed to hold NSStrings. As this value will not be accepted by Capture, the JUMP for iOS library will filter the array, accepting only the correct element type, before replacing it on Capture.
